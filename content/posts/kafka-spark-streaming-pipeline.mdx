## Introduction

Real-time data pipelines are no longer a luxury — they're a core requirement for modern data-driven applications. In this post, I'll walk through building a production-grade streaming pipeline using **Apache Kafka** and **Apache Spark Structured Streaming**.

## Architecture Overview

Our pipeline follows the classic Lambda-lite pattern:

1. **Producers** publish events to Kafka topics
2. **Spark Streaming** consumes and transforms events
3. **Delta Lake** stores processed data
4. **Downstream consumers** read from Delta

## Setting Up Kafka

First, start a local Kafka cluster using Docker:

```yaml
version: '3'
services:
  zookeeper:
    image: confluentinc/cp-zookeeper:7.4.0
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181

  kafka:
    image: confluentinc/cp-kafka:7.4.0
    depends_on:
      - zookeeper
    ports:
      - "9092:9092"
    environment:
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
```

## Spark Structured Streaming Consumer

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import from_json, col
from pyspark.sql.types import StructType, StringType, TimestampType, DoubleType

spark = SparkSession.builder \
    .appName("RealTimePipeline") \
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .getOrCreate()

schema = StructType() \
    .add("user_id", StringType()) \
    .add("event_type", StringType()) \
    .add("timestamp", TimestampType()) \
    .add("value", DoubleType())

df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9092") \
    .option("subscribe", "events") \
    .load() \
    .select(from_json(col("value").cast("string"), schema).alias("data")) \
    .select("data.*")

query = df.writeStream \
    .format("delta") \
    .outputMode("append") \
    .option("checkpointLocation", "/tmp/checkpoints/events") \
    .start("/data/events")

query.awaitTermination()
```

## Key Considerations

- **Exactly-once semantics**: Use Kafka transactions + Delta Lake ACID guarantees
- **Backpressure**: Configure `maxOffsetsPerTrigger` to control throughput
- **Schema evolution**: Delta Lake handles schema changes gracefully
- **Monitoring**: Track `inputRowsPerSecond` and `processedRowsPerSecond`

## Conclusion

This architecture scales horizontally — add Kafka partitions and Spark executors as your throughput grows. The combination of Kafka's durability and Delta Lake's ACID properties gives you both speed and reliability.
