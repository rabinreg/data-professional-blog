## Why dbt Changed Data Engineering

dbt (data build tool) brought software engineering best practices to SQL-based data transformations. Version control, testing, documentation, and modular design — all in the analyst's language.

After running dbt across several production environments, here are the patterns that work and the traps to avoid.

## Project Structure That Scales

```
models/
  staging/        # 1:1 with source tables, light cleaning only
  intermediate/   # business logic joins, kept private
  marts/          # final business-facing models
    finance/
    marketing/
    product/
```

The key insight: **staging models should be cheap**. One source table, one staging model. No joins. No business logic. Just clean, rename, cast.

## Testing Strategy

Don't just test `not_null` and `unique`. Test what matters to the business:

```yaml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'delivered', 'cancelled']
      - name: revenue_usd
        tests:
          - not_null
          - dbt_utils.expression_is_true:
              expression: ">= 0"
```

## Incremental Models Done Right

```sql
{{
  config(
    materialized='incremental',
    unique_key='event_id',
    on_schema_change='sync_all_columns'
  )
}}

select
  event_id,
  user_id,
  event_type,
  occurred_at
from {{ ref('stg_events') }}

{% if is_incremental() %}
  where occurred_at > (select max(occurred_at) from {{ this }})
{% endif %}
```

Watch out for **late-arriving data** — consider a lookback window instead of strict `max()`.

## CI/CD with Slim CI

Avoid rebuilding everything on every PR. Use dbt's `state:modified` selector:

```bash
# Only run models changed in this PR
dbt run --select state:modified+ --defer --state ./prod-artifacts
dbt test --select state:modified+ --defer --state ./prod-artifacts
```

## Top Pitfalls

1. **Fan-out joins in staging** — catches you in perf reviews
2. **No freshness tests** — data goes stale silently
3. **Skipping `dbt docs generate`** — technical debt compounds fast
4. **Giant CTEs** — break them into intermediate models

## Conclusion

dbt is transformative when used with discipline. Invest in your project structure early, test aggressively, and treat your SQL like the production code it is.
